Particiones de arranque

boot, vendor_boot e init_boot

La organización cambia según versión y dispositivo. En Android 12, boot contiene GKI y generic ramdisk; en Android 13+, el generic ramdisk se mueve a init_boot.

Android 11/12

  • boot: kernel/GKI y generic ramdisk.
  • vendor_boot: vendor ramdisk y datos dependientes del proveedor.
  • vbmeta: metadatos AVB.

Android 13+

  • boot: GKI kernel.
  • init_boot: generic ramdisk.
  • vendor_boot: vendor ramdisk, DTB según versión.

Android Init

Del primer proceso de userspace a los servicios Android

El lenguaje Android Init se compone de Actions, Commands, Services, Options e Imports. Los scripts son lineales y se disparan mediante triggers.

Actions

Bloques que se ejecutan cuando se activa un trigger, por ejemplo on early-init, on init o on boot.

Services

Procesos gestionados por init. Incluyen binario, argumentos, clase, usuario, grupo, sockets y comportamiento al reinicio.

Imports

Permiten cargar otros ficheros .rc, como scripts de zygote, vendor o hardware.

Zygote

El punto de partida de los procesos Android de alto nivel

Zygote es iniciado por init mediante app_process/app_process64. Prepara ART, abre sockets y arranca System Server.

¿Por qué existe Zygote?

Permite que los procesos Android compartan una base común cargada previamente. Esto reduce coste de arranque y aprovecha copy-on-write.

System Server

Desde Zygote arranca System Server, que inicializa servicios centrales del framework como ActivityManager, PackageManager y WindowManager.

Comandos útiles

Observación del proceso en un entorno Android

Comandos orientados a comprender el arranque en un dispositivo o entorno de desarrollo. Su disponibilidad depende del build, permisos y modo de depuración.

Parámetros de arranque

adb shell cat /proc/cmdline
adb shell cat /proc/bootconfig

Servicios gestionados por init

adb shell getprop init.svc.zygote
adb shell getprop | grep '^\[init.svc'

Procesos principales

adb shell ps -A | grep zygote
adb shell ps -A | grep system_server

Tiempo de arranque

adb logcat -b events | grep boot
adb shell dmesg | head
